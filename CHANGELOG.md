### Added — 2026-07-14 (AI Gateway boundary, respecting OTP/session gating)

- **Resolves the architecture conflict flagged in the previous round**: `SYSTEM_ARCHITECTURE.md` §6/§8 requires all AI access go through an internal AI Gateway, never call OpenAI directly from production. New `src/modules/smart-clinic-assistant/ai/` boundary module — `ai-gateway-client.ts` (single exported entry point, `callAiGateway(task, input, {sessionVerified})`), `config.ts` (`isGatewayConfigured`, `isDirectOpenAiDevAllowed` — the latter checks `NODE_ENV === "production"` first and returns `false` unconditionally, structurally blocking any production direct-OpenAI path regardless of the dev flag's value), `types.ts` (the full `classify_assistant_message`/`generate_lead_summary` task contract), `dev-openai-fallback.ts` (the only file in the module that knows OpenAI exists — local dev only, requires `ALLOW_DIRECT_OPENAI_IN_DEV=true` + a key, never reachable in production).
- Deleted the round-D direct-call files (`server/ai/openai-client.ts`, `server/ai/config.ts`) — superseded, no longer referenced anywhere (`grep` confirmed clean).
- `intent-detector.ts`/`lead-summary.ts` refactored to call `callAiGateway` against the new richer input/output shapes (`confidence`, `safetyFlag`, `suggestedNextStep`, `leadPriority`, etc.) instead of talking to OpenAI directly. `detectFreeTextIntent`/`generateLeadSummary` now take an explicit `sessionVerified` boolean, threaded through (not re-derived) from `interpret-free-text.ts`/`submit-booking-request.ts`'s already-computed `isSessionVerified` result — verification is checked at three independent points (drawer's `runGated`, the server entry point, and `callAiGateway` itself) before any AI call, gateway or dev-fallback alike.
- Per the brief's explicit no-AI-trigger list and cost ceiling: normal menu-driven flow makes zero AI calls; worst case (unmatched free-text + completed booking) makes at most 2 — unchanged from the previous round, now enforced through the gateway boundary rather than a direct client.
- `.env.example` updated: `INTERNAL_AI_GATEWAY_URL`/`INTERNAL_AI_GATEWAY_TOKEN` (unset in this repo — no such service is deployed anywhere yet), `ALLOW_DIRECT_OPENAI_IN_DEV=false` (explicitly documented as structurally ignored in production), `OPENAI_API_KEY`/`OPENAI_MODEL` re-scoped as dev-fallback-only.
- Verified via Playwright (foreground agent run) with nothing configured: deterministic booking path unaffected, local-keyword free-text works without any AI call, an ambiguous free-text message degrades gracefully in ~160ms with zero `[ai-gateway]`/`[dev-openai-fallback]` log lines (confirming a true short-circuit at the "not configured" check, not a silent timeout), full booking completes to confirmation, zero console errors, zero secrets in logs. Separately verified, with a dev server restarted under `ALLOW_DIRECT_OPENAI_IN_DEV=true` + a fake key, that the dev-fallback path is genuinely attempted (server log shows `[dev-openai-fallback] non-2xx response 401` — status code only, never the key) — confirming the "verified + configured" path is really wired, not just quietly doing nothing either way.
- `AI_USAGE_NOTES.md` rewritten for the gateway-first architecture: transport-selection logic, the two-gates model (verified session + configured transport), the full gateway request/response contract, and an explicit privacy-safeguards summary.

### Added — 2026-07-14 (mobile OTP verification gates AI + booking, docs/adr/0007)

- **New real DB models**: `OtpCode` (mobile, `codeHash` only — raw code never stored, `purpose`, `expiresAt`, `consumedAt`, `attempts`) and `AssistantSession` (mobile, `verifiedAt`, `locale`), both tenant-scoped (`clinicId` + index) per `DATABASE_GUIDE.md` §1. New migration hand-authored to match Prisma's own conventions exactly (verified byte-for-byte against a fresh `prisma migrate diff --from-empty` of the full current schema — no live DB available to generate an incremental diff directly).
- `server/otp/` — new: `otp-crypto.ts` (Node's built-in `scrypt`, no bcrypt/argon2 dependency added — justified in docs/adr/0007), `otp-repository.ts` (Prisma-backed, clinicId-scoped), `sms-provider.ts` (no-op placeholder, mirrors `application/sms-events.ts`'s existing pattern), `session-guard.ts` (`isSessionVerified` — the one function every gated Server Action calls).
- `server/request-otp.ts` / `server/verify-otp.ts` — new Server Actions. **Real DB-backed path** when a database and SMS provider are both configured (neither is, in this environment); **honest "unavailable" fallback** (the exact required Persian message) when they're not, in production; a narrow, explicitly-labeled **dev bypass** (fixed code `000000`, zero DB writes, structurally blocked by `NODE_ENV === "production"` on both the issuing and checking side) for local verification — no in-memory store was used as a stand-in production solution, per explicit instruction.
- **Four gate points**, added to `AssistantDrawer` via one `runGated` chokepoint: submitting the free-text input, completing triage, choosing "general consultation," and the final booking submit (defense-in-depth re-checked server-side in `submit-booking-request.ts` and `interpret-free-text.ts` regardless of client state — a client cannot fake verification). Everything else (browsing, starting triage, viewing service cards, informational steps, language switching) stays fully open, per the explicit allow-list.
- New `PhoneVerificationStep` component and `"phone_verification"` `AssistantStep`. **Verification interrupts, never restarts, the flow**: the drawer captures the exact blocked action in a ref and replays it — with the freshly-issued token, not a stale closure-captured one — on success. `GeneralStep`'s free-text input/loading/error state moved up to the drawer for this reason (it needs to survive `GeneralStep` unmounting while verification is shown). The verified mobile pre-fills (doesn't lock) `ContactCaptureStep`'s mobile field.
- Verified live end-to-end via Playwright: pre-verification browsing fully unrestricted; both gate paths (free-text ask, triage completion) correctly interrupt and — this was the critical thing to get right — correctly **resume the exact interrupted destination** after verification (landed on the routed cost-question screen / the contact-info form with mobile pre-filled, never a reset to the blank menu); a full booking completes without re-prompting once verified; server log confirms only the intentional, clearly-labeled dev-bypass hint is ever logged — no real OTP code or hash appears in any log line.
- `AI_USAGE_NOTES.md` updated: unverified/anonymous traffic now cannot reach OpenAI at all (a real secondary effect of this gate, not just a UX gate).

### Added — 2026-07-14 (cost-controlled OpenAI integration, AI_USAGE_NOTES.md)

- **Architecture flag**: this brief describes calling OpenAI directly, which conflicts with `SYSTEM_ARCHITECTURE.md` §6/§8's confirmed rule (never call OpenAI/Anthropic directly from Iran-hosted production infra — always through the internal AI Gateway, a separate service that doesn't exist in this repo yet). Built behind one swap point (`server/ai/openai-client.ts`) instead of silently ignoring or silently violating the rule — every other AI file calls only that file's two exports, so routing through the real gateway later is a one-file change. - `src/modules/smart-clinic-assistant/server/ai/` — new: `config.ts` (env reading), `openai-client.ts` (the one direct-call site, 8s timeout, never throws), `local-intent-matcher.ts` (zero-cost fa/en/ar keyword + service-name matching), `intent-detector.ts` (local-first, AI-fallback, single compact call classifies-or-answers), `response-cache.ts` (in-process, capped, TTL'd — no new infra dependency), `lead-summary.ts` (one call per completed booking, PII-minimized payload).
- `interpret-free-text.ts` — new Server Action, the only entry point into any AI code from the UI.
- `GeneralStep` gained a free-text input below its existing 5 buttons — the only new UI surface added; every other step is unchanged. New `"qa_response"` `AssistantStep` reuses the existing `InfoStep` component (no new step component).
- `submit-booking-request.ts` now generates an internal (staff-only, never patient-facing) lead summary exactly once per completed booking — logged only (no `Lead.aiSummary` DB column added; a real schema change wasn't authorized in this pass, flagged as deliberate follow-up work).
- **Verified in this environment without a real API key** (none available — same "offline-safe, verified as far as possible" pattern as the DB persistence work): confirmed local keyword matches resolve in ~50-80ms with zero network/log activity, an unmatched free-text message degrades to the inline "couldn't understand" message without crashing, and a full booking flow completes normally with the lead-summary call failing silently (no key configured). The actual OpenAI call path itself (network request, response parsing) is implemented and type-checked but not exercised end-to-end here — real verification needs a real `OPENAI_API_KEY`.
- See `AI_USAGE_NOTES.md` for the full policy: exactly which actions call OpenAI (2 of the many), which never do (the entire deterministic booking path), the 3-tier fallback, and exactly what data is/isn't sent.
