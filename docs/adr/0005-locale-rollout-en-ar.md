# 0005. Locale rollout: English and Arabic homepage support begins

Status: Accepted
Date: 2026-07-13

## Context

`docs/adr/0002-fa-first-locale-scope.md` deliberately launched with Persian-only content, explicitly framing `en`/`ar` as "Phase 2 of this locale rollout" once real content arrived. Hamid has now confirmed the product decision: the site must support Persian, English, and Arabic. This ADR is that Phase 2 starting — it does not reverse 0002's reasoning, it fulfills the continuation 0002 already anticipated.

Scope constraint carried over from Hamid's brief: no homepage redesign, and per `CLAUDE.md`'s standing section-by-section design rule, no new unilateral visual design either. Everything below is a **content/data-plumbing change** — the exact same components, same visual design, fed by a different dictionary object — not a new layout for `en`/`ar`.

## Decision

- **Persian remains the primary source of truth.** New copy is written in Persian first (as it always has been — see `fa.ts`'s own doc-comment), then translated. `fa`/`ar` are RTL, `en` is LTR — unchanged from `i18n/locales.ts`'s existing `LOCALE_DIRECTION` map, already correct.
- **`getDictionary(locale)`** (`src/i18n/get-dictionary.ts`) is the one new seam: given a `Locale`, returns the right dictionary object. Components that already accept a `dict` prop (e.g. `SiteFooter`) or are converted to accept one in this round (`SiteHeader`, `MobileMenu`) are wired through it from `layout.tsx`/`page.tsx`.
- **Full dictionary parity is explicitly NOT attempted in this pass.** `fa.ts` is ~600 lines covering 11 homepage sections, the full Smart Clinic Assistant flow (9 services × triage questions), FAQ copy, patient stories, etc. Translating all of it to a premium, non-machine-garbled standard in one pass is not realistic, and Hamid's own brief explicitly permits a critical-path-first partial rollout ("if full homepage translation is too large... translate the critical public UI first... report what remains"). `en.ts`/`ar.ts` (`src/i18n/dictionaries/`) therefore cover: header, footer, hero, Smart Clinic Assistant entry copy + service names + core flow labels, and primary CTAs — see those files' own doc-comments for the exact key list and CHANGELOG.md for what's deliberately deferred.
- **`en.ts`/`ar.ts` are typed against narrow per-section interfaces** (`src/i18n/dictionary-types.ts`: `HeaderDictionary`, `FooterDictionary`, `HomepageHoldingDictionary`, etc.), not against `Dictionary = typeof fa` (fa.ts's full shape). Forcing full-shape parity would mean either fabricating translations for content that doesn't exist in en/ar yet, or littering `as Dictionary` casts to lie about it — both worse than an honest partial type. `fa.ts` satisfies the same narrow interfaces for content it actually has (`HeaderDictionary`, `FooterDictionary`, via `satisfies`) — the holding-page copy (`HomepageHoldingDictionary`) only exists where it's needed, `en`/`ar`, since `fa` never renders a holding page.
- **The homepage body for `en`/`ar` is a minimal, honest holding view** (`src/app/[locale]/page.tsx`), not the full 11-section Persian homepage — because most sections aren't translated yet (see above), and because fabricating a full page from partial content would itself be a design decision no one asked for. It reuses the same design tokens/typography as the rest of the site (not a new visual system), states plainly that full content is coming, and offers the one thing that's fully functional regardless of language: the Smart Clinic Assistant CTA. This replaces the previous behavior (`notFound()` → generic Next.js 404) for `en`/`ar`, which was strictly worse: indistinguishable from a broken link, and gave the `LanguageSwitcher` nothing real to switch to (that component's own doc-comment already flagged this as the accepted-but-known state under 0002).
- **Header nav anchors (`#why-dr-sadighi`, `#services`, etc.) are left unchanged for `en`/`ar`.** They point at section IDs that only exist on the `fa` homepage. On the `en`/`ar` holding page they are inert (no matching ID, so the browser simply does nothing) rather than broken — an acceptable, documented gap until per-section translation lands, not silently papered over.
- **The Smart Clinic Assistant drawer itself is NOT locale-wired in this pass.** It is mounted globally in `layout.tsx` and reads `fa.assistantFlow` as a module-level constant throughout its step components; converting that to be locale-aware is a separate, larger refactor (every step component, the reducer, validation messages) explicitly out of scope for "start en/ar homepage dictionaries." An `en`/`ar` visitor who opens the assistant today still sees a Persian-language flow — flagged as known remaining work, not a silent gap.

## Consequences

- `docs/adr/0002-fa-first-locale-scope.md` is annotated (not deleted or rewritten) as partially superseded by this ADR — its historical reasoning for launching `fa`-only first is still accurate and worth keeping.
- `en`/`ar` visitors now get a real, correctly-directioned (LTR/RTL), professionally-translated header, footer, and holding homepage instead of a 404 — a meaningful improvement, not full parity.
- Remaining translation work (all 11 homepage sections' full copy, FAQ answers, patient stories, case gallery, knowledge center, the assistant flow's triage questions) is real, sizable, and explicitly not started — tracked here rather than silently implied "done" by the existence of `en.ts`/`ar.ts`.
- Wiring the Smart Clinic Assistant drawer itself to locale is a follow-up ADR-worthy piece of work (it touches the reducer, every step component, and validation error messages — a real architectural change, not a copy swap) once full section translation is prioritized.
